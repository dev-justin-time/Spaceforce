1. System Architecture

Spaceforce Squadron is built on a modular Component-Based Simulation Engine utilizing Three.js for high-fidelity 3D rendering and WebsimSocket for real-time synchronization. The architecture prioritizes data persistence and cross-platform performance.

State Layer

Redux-style central store (src/store.js) managing score, health, and player loadout state. Unidirectional data flow ensures predictable UI updates.

Persistence Layer

Database API (src/database.js) mapping local session events to encrypted Websim collections for cross-session inventory and reputation storage.

Simulation Layer

The Game loop (src/game.js) synchronizes the World, EntityManager, and Input systems at a consistent 60fps target.

2. Networking & Multiplayer

The neural-link protocol utilizes a low-latency presence broadcasting system. To optimize bandwidth, only critical positional vectors and status flags are transmitted via the sub-space channel.

  • Sector Separation: Pilots are filtered by levelId presence keys to prevent cross-sector interference.
  • Collision Reconciliation: Client-side authoritative logic for local hits, with server-side event logging for "Uni" credit validation.

3. Gameplay Systems

World & Levels (src/world.js, src/data/levels.js)

  • World.loadLevel(id) clears scene, sets atmosphere (lighting, fog, background) and terrain material.
  • Pano backgrounds 1–8 are selected randomly per load via setupAtmosphere().
  • update(dt, score) scrolls chunks, introduces lateral “curve” at high score, and animates background overlays.
  • levels.js defines sector, thresholds, difficulty, rewards and mode flags (PvP/PvAI).

Player & Loadouts (src/player.js)

  • Uses shared s.gltf ship model when available; falls back to procedural ship meshes by hull type.
  • applyLoadout(inventory) parses item effect strings and ship hulls to compute multipliers.
  • Weapon states: weaponLevel (spread), enableRapidFire(), primary/secondary cooldowns.
  • update(dt,input) handles bounds, tilt/bank, and returns a firing state for the entity system.

Entities & Collisions

  • EntityManager spawns: bullets, missiles, coins (1/5/10/50 UNI tiers), powerups, asteroids, alien ships.
  • Enemy patterns are centralized in EnemyFactory, e.g. spawnGrey, spawnDogfighter, spawnPredator.
  • CollisionSystem.checkCollisions() handles: coin pickup, powerup pickup, bullets vs enemies, player vs enemies, player vs enemy bullets.
  • Damage/score callbacks are passed from Game so collisions remain stateless.

Spawn Logic (Game.animate())

  • Per‑frame: get dt, update world, timers, entities, then camera, then render.
  • Enemy spawn behavior branches:
    • Dogfight levels (90–94) – dogfighters only, tuned counts/intervals.
    • PvP arena (92) – AI dogfighters only when no human peers.
    • Standard levels – mix of asteroids, fighter squads, “Greys”, plus level‑specific alien types.
  • Powerups and coins spawn on independent timers with random placement.

4. Controls & Input

All input is centralized in src/input.js, which merges keyboard, mouse, gyro, and touch joysticks into a single { move, shoot, shootSecondary, autoShoot } struct consumed by Player.update().

  • Desktop: WASD/Arrows for movement, Space / Mouse Left for primary, Mouse Right for secondary when mouseControls is enabled.
  • Mouse aim: world‑space P‑controller that drives move towards cursor within player bounds.
  • Mobile: nipple.js static sticks on left/right zones when touchControls is on.
  • Gyro: optional tilt control with permission gating; blends with other inputs based on store flags.

5. UI & Screens

Template Files (src/ui/)

  • hangar.html – mode/sector selection, mission list, header icons.
  • game-hud.html – in‑flight HUD, score, shields, camera & settings.
  • game-over.html – post‑mission stats, credits earned, restart to hangar.
  • market.html – SpaceForce systems depot & player inventory sell UI.
  • leaderboard.html, profile.html, codex.html, settings.html – meta screens.

UI Manager (src/ui.js)

  • loadTemplates() injects all UI partials into #ui-layer from src/ui/*.html.
  • Wires hangar buttons, codex tabs, modals, settings toggles, chat focus, and joystick visibility.
  • Subscribes to store updates to render score, health, loadout, and game over overlay.
  • Loads pilot avatar and stats from Websim user API + database.

6. Networking & Multiplayer

Presence & Remote Ships

  • MultiplayerManager wraps a WebsimSocket room.
  • updatePlayerPresence() broadcasts x, y, status, alive.
  • syncRemotePilots() mirrors other clients as simple ship meshes inside the main scene.
  • inflictDamage() sends presence update requests to apply PvP damage.

Chat & AI Helper

  • Chat messages are stored in chat collection; MultiplayerManager keeps last 50.
  • /ai or @helper triggers a Websim LLM completion with recent chat as context.
  • AI responses are posted back into chat as system messages prefixed with [AI].
  • Scores are broadcast via scores collection and rendered in both minimal HUD leaderboard and full modal.

7. Economy, Market & Codex

Economy

  • Uni credits are stored per account and synced into the client store via HangarManager.loadUserData().
  • db.addTransaction() updates account balance and logs a transaction record.
  • Coins in‑game directly call ADD_CREDITS and persist via loot transactions.
  • AccountantAgent uses Websim LLM to generate a short “Ledger” report for the hangar footer.

Market & Codex

  • MarketManager renders flip‑cards with art, classification, specs, and purchase flow.
  • db.seedMarket() populates listings from militarySystems with dynamic pricing (TOP SECRET / Ancient Tech high cost).
  • Codex UI shows the same lore items as static flip‑cards organized into Ranks, Systems, Factions, Xenology tabs.
  • Inventory can be re‑listed into the market with custom price; purchased systems are added back into inventory.

8. Visual Tuning & Brightness

Brightness is primarily controlled via RendererManager and level lighting.

  • renderer.toneMappingExposure = 0.7 reduces global intensity vs default.
  • UnrealBloomPass uses threshold=0.35, strength=0.7, radius=0.45 to avoid blown‑out highlights.
  • Per‑level lighting in World.setupLighting() can be adjusted for specific scenes that are “too bright”.

9. Extension Notes

  • Add new levels by extending levels.js and handling lighting/environment in World.
  • New enemies should be implemented in EnemyFactory and wired into spawn logic in Game.animate().
  • New items go into src/data/items.js; pricing behavior is derived in db.seedMarket() and bonuses parsed in Player.applyLoadout().
  • To tweak controls defaults, adjust the auto‑enable logic in UIManager.init() and InputManager.
